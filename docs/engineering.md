# dlite Web Components — Engineering Approach

## Overview

`web-components-dlite` is a library of Lit-based web components styled exclusively with design tokens from `style-dictionary-dlite-tokens`. The components are framework-agnostic: they work in React, Astro, Enhance, vanilla HTML, and any environment that supports Custom Elements.

---

## Architecture

### Design Token Integration

Every visual property — color, spacing, typography, border-radius, elevation, duration — comes from CSS custom properties generated by Style Dictionary. Components never hard-code visual values.

```
--tk-dlite-semantic-color-action-primary   → focus rings, active states
--tk-dlite-semantic-color-feedback-danger   → error borders, error text
--tk-dlite-semantic-typography-font-heading → sans-serif (Plus Jakarta Sans)
--tk-dlite-semantic-typography-font-body    → serif (Source Serif 4)
```

### Shadow DOM

All components use Shadow DOM for style encapsulation. This means:

- Component styles never leak into the consumer's page
- Consumer styles never accidentally break a component
- Slotted content (light DOM children) inherits the consumer's styles
- Native events from inner elements may not cross the shadow boundary (see Events below)

## Event Strategy

### The Problem

Web components use Shadow DOM, which affects how events propagate:

- **`input` events** from native `<input>` / `<textarea>` elements have `composed: true` — they cross the shadow boundary automatically.
- **`change` events** from native `<input>` / `<select>` / `<textarea>` elements have `composed: false` — they do **not** cross the shadow boundary.

This creates an inconsistent experience for consumers: `onInput` works on `<dl-input>` but `onChange` silently does nothing (the native `change` never reaches the host element).

### The Solution: Dual-Event Dispatch

Every event-emitting component dispatches **two events** for each action:

1. **Semantic custom event** (`dl-change`, `dl-input`, `dl-close`, `dl-dismiss`) — prefixed, always available, carries `detail` payload.
2. **Native-compatible event** (`change`, `input`, `close`) — same name as the native event, dispatched as `CustomEvent` with `composed: true` and the same `detail` payload.

For components that wrap native form elements, the native event from the inner element is **stopped** (`e.stopPropagation()`) so consumers receive exactly **one** clean event of each type from the host — no double-firing.

### Event Contract by Component

| Component      | Semantic Event | Native-Compatible Event | Detail              |
| -------------- | -------------- | ----------------------- | ------------------- |
| `dl-input`     | `dl-input`     | `input`                 | `{ value: string }` |
| `dl-input`     | `dl-change`    | `change`                | `{ value: string }` |
| `dl-textarea`  | `dl-input`     | `input`                 | `{ value: string }` |
| `dl-textarea`  | `dl-change`    | `change`                | `{ value: string }` |
| `dl-select`    | `dl-change`    | `change`                | `{ value: string }` |
| `dl-checkbox`  | `dl-change`    | `change`                | `{ checked: boolean }` |
| `dl-toggle`    | `dl-change`    | `change`                | `{ checked: boolean }` |
| `dl-tabs`      | `dl-change`    | `change`                | `{ value: string }` |
| `dl-dialog`    | `dl-close`     | `close`                 | —                   |
| `dl-alert`     | `dl-dismiss`   | `close`                 | —                   |

### Framework Usage

**React** — use standard `onChange` / `onInput` / `onClose` props:

```tsx
<dl-input onInput={(e) => setValue(e.detail.value)} />
<dl-select onChange={(e) => setChoice(e.detail.value)} />
<dl-tabs onChange={(e) => setTab(e.detail.value)} value={tab}>…</dl-tabs>
```

**Vanilla JS** — use either event name:

```js
document.querySelector('dl-input').addEventListener('input', (e) => {
  console.log(e.detail.value);
});
// or the semantic variant
document.querySelector('dl-input').addEventListener('dl-input', (e) => {
  console.log(e.detail.value);
});
```

**Astro / Enhance / SSR** — same as vanilla; attach listeners after hydration.

### Why Both Events?

- **Native-compatible events** (`change`, `input`, `close`) work with framework prop bindings (`onChange`, `onInput`, `onClose`) out of the box — no refs or imperative code needed.
- **Semantic events** (`dl-change`, `dl-input`, `dl-close`, `dl-dismiss`) are explicit, avoid collisions with other libraries, and make it clear the event came from a dlite component when debugging.

## Shadow DOM Limitations

### dl-table Striped Rows

`dl-table` sets a CSS custom property `--dl-table-stripe-bg` when `striped` is true, but **cannot style slotted `<tr>` elements** from inside Shadow DOM (slotted content retains light-DOM styling). Consumers must add companion CSS in their own stylesheet:

```css
dl-table[striped] tr:nth-child(even) {
  background: var(--dl-table-stripe-bg, rgba(0, 0, 0, 0.03));
}
```

### Font Loading

Components reference font-family tokens but do **not** load fonts themselves. The consuming application must load the required Google Fonts (or equivalent):

- **Plus Jakarta Sans** — headings, tab buttons
- **Source Serif 4** — body text
- **Source Code Pro** — code/monospace

## React Type Definitions

The build generates `dist/react.d.ts` which provides JSX IntrinsicElements types for all 20 components. This gives TypeScript consumers autocomplete and type-checking for properties and events.

Generated by `scripts/generate-react-types.ts` during `npm run build`.
